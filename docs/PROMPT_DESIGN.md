# LLMプロンプト設計書

## 概要
GaggiMate連携アプリでは、LM Studio（ローカルLLM）を2つの用途で使用する:
1. **ショット改善提案** — 抽出データ + ユーザーフィードバック → 次のショットの改善アクション
2. **レシピカスタマイズ** — ユーザーの要望 → GaggiMateプロファイルJSON生成

推奨モデル: Qwen2.5:14B（RTX 4080で快適動作）

---

## 1. ショット改善提案プロンプト

### システムプロンプト
```
あなたはエスプレッソ抽出の専門家です。
ユーザーのショットデータ（圧力・温度・フロー・時間・収率）とフィードバック（味の感想）を分析し、
次のショットで改善すべき具体的なアクションを3つ以内で提案してください。

提案は以下の形式で:
1. 【アクション】具体的な操作（例: グラインドを1クリック細かく）
2. 【理由】なぜそう提案するか

使用グラインダー: コマンダンテ C40（クリック数で管理）
マシン: Gaggia Classic E24 + GaggiMate Pro（圧力プロファイル制御可能）
```

### ユーザーメッセージテンプレート
```
## ショットデータ
- 抽出時間: {duration}秒
- ドーズ: {dose_g}g → 抽出量: {yield_g}g（収率: {yield_ratio}x）
- 平均圧力: {avg_pressure}bar / ピーク: {peak_pressure}bar
- 平均温度: {avg_temp}°C
- 平均フロー: {avg_flow}ml/s
- コマンダンテ クリック数: {clicks}（設定されている場合）
- 豆: {bean_info}（設定されている場合）

## 自動検知された問題（Pythonルールベース分析の結果）
- ⚠️ {flag_1}
- ⚠️ {flag_2}
...

## 過去ショットとの比較（同じ豆の直近5件）
- 25秒 / 2.0x / ★4 / バランス良い
- 22秒 / 1.8x / ★3 / 酸味が強い
...

## ユーザーフィードバック
{feedback}
```

### 設計ポイント
- **数値分析はPythonが担当**: 収率計算、異常検知フラグはLLMに渡す前にPythonで算出
- **LLMは言語化と推論を担当**: 「酸っぱい + 18秒 → グラインドを細かく」のような因果推論
- **過去データで文脈を提供**: 同じ豆の過去ショットがあれば比較材料として渡す
- **具体的なアクション**: 「1クリック細かく」のように数値で指示

### 異常検知フラグ（Pythonルール）
| 条件 | フラグメッセージ |
|---|---|
| 抽出時間 < 目標-5秒 | 「抽出時間が目標よりN秒短い（チャネリングの可能性）」 |
| 抽出時間 > 目標+8秒 | 「抽出時間が目標よりN秒長い（挽き目が細すぎる可能性）」 |
| 収率 < 1.5x | 「収率が低い — ドーズ過多 or 抽出不足」 |
| 収率 > 3.0x | 「収率が高い — 過抽出の可能性」 |
| ピーク圧力 > 10.5bar | 「ピーク圧力が高い — パック詰まり」 |
| 平均圧力 < 6.0bar | 「平均圧力が低い — 挽き目が粗い可能性」 |

## 2. レシピカスタマイズプロンプト

### システムプロンプト
```
あなたはエスプレッソレシピの専門家です。
GaggiMate Proのプロファイル形式（JSON）でレシピを提案してください。

プロファイル構造:
- label: レシピ名
- type: "pro"
- temperature: ボイラー温度（°C）
- phases: [{name, phase, valve, duration, pump: {pressure, flow}, transition: {type, duration}, targets: {pumped, weight}}]

過去のショットデータとユーザーの要望に基づいて、最適なプロファイルを生成してください。
レシピJSONは```json ... ```で囲んで出力してください。
```

### ユーザーメッセージテンプレート
```
## リクエスト
{user_request}

## ベースレシピ（選択された場合）
```json
{base_recipe_json}
```

## 過去ショット（直近5件）
- 28秒 / 2.0x / ★4 / バランス良い
- 22秒 / 1.8x / ★3 / 酸味が強い
...
```

### 設計ポイント
- **GaggiMateプロファイル形式を厳守**: JSONの構造をシステムプロンプトで明示
- **ベースレシピ**: お気に入りレシピがあればベースとして渡す → 微調整を促す
- **過去データ**: 過去のショットを渡すことで、ユーザーの好みを推測可能

## 3. プロンプトテンプレート管理

### テンプレート保存先
`backend/app/prompts/` ディレクトリにテキストファイルとして管理:
- `shot_improvement_system.txt` — 改善提案システムプロンプト
- `shot_improvement_user.txt` — 改善提案ユーザーテンプレート
- `recipe_customize_system.txt` — レシピカスタマイズシステムプロンプト
- `recipe_customize_user.txt` — レシピカスタマイズユーザーテンプレート

### テンプレート変数
テンプレート内で `{variable_name}` 形式のプレースホルダーを使用。
Python側で `template.format(**context)` で展開する。

### REST API
- `GET /api/prompts` — テンプレート一覧
- `GET /api/prompts/{name}` — テンプレート取得
- `PUT /api/prompts/{name}` — テンプレート更新（カスタマイズ）

## 4. パラメータ設定

| パラメータ | 値 | 理由 |
|---|---|---|
| temperature | 0.7 | 具体的な提案を出しつつ、柔軟性を確保 |
| max_tokens | 1024（改善提案）/ 2048（レシピ） | 提案は簡潔、レシピはJSON含むため長め |
| model | config.jsonの `lm_studio_model` | LM Studioでロード中のモデルを使用 |

## 5. 品質向上のためのヒント

### 良い結果を出すために
- **具体的なフィードバックを書く**: 「まあまあ」より「酸味が強くて、ボディが薄い」
- **グラインド情報を入力する**: クリック数があるとLLMが具体的な調整値を出せる
- **豆情報を登録する**: 同じ豆の過去データが比較に使われる

### LLMの限界
- **物理的な制約**: LLMは機械の物理特性を完全には理解しない
- **個人の好み**: 味の好みは主観的 → フィードバックの蓄積で精度が上がる
- **レシピJSON**: 生成されたJSONは必ず確認してからGaggiMateに送信する
